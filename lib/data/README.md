# Data Layer — contract

Every page/component gets data through `getDataSource()` from `lib/data/index.ts`.
Never import `GoogleSheetsDataSource` or `QBODataSource` directly outside that factory —
that's the whole point of the `DataSource` interface: swapping a client's backend later is
a one-line change in `client.config.ts`, not a rewrite of the dashboard.

## Files

- `types.ts` — shapes every source must return (`MetricRow`, `FinancialStatementRow`, `CustomReportData`...).
  UI components only ever see these, never a raw sheet row or raw API response.
- `source.ts` — the `DataSource` interface itself.
- `sources/googleSheets.ts` — implementation. Full sheet contract (fixed tab names,
  fixed header row per tab) is documented at the top of that file — read it before
  connecting a new client's sheet.
- `sources/qbo.ts` — **not implemented yet**, stub only. Wire it up when QBO becomes a
  real backend for a client; until then `client.config.ts` should stay on `googleSheets`.
- `index.ts` — the factory. Picks the implementation based on `client.config.ts`'s
  `dataSource.type`.

## Where the Dashboard tab's narrative comes from

The Executive Summary text on the Dashboard tab (`getDashboardSummary()`) is **not**
computed by this app and never generated by an LLM call at page-render time. It's
authored offline — an FM runs a Claude analysis skill (the same close-cycle workflow
that already produces internal `YYYY_MM_Summary` insights) against the raw transactions
for the period, and the resulting sentence is written into the `Dashboard_Data` tab.
This app only reads and displays it. Everything else on the Dashboard tab (summary
cards, trend charts) IS computed live, deterministically, from the `PL`/`CF`/`BS`
statements already fetched via `getStatement()` — see `lib/calc/dashboardMetrics.js`.

## Adding a new client on Google Sheets

1. Build the client's Sheet using the fixed tabs: `KPI_Report`, `PL`, `CF`, `BS`,
   `Dashboard_Data`, `Custom_Reports_Index` (+ one tab per custom report, referenced
   from the index).
2. Create a **dedicated** Google Service Account for this client (never reuse another
   client's — see `fuel-vercel-dashboard-design-rules.md` for why) and share the Sheet
   with it as Viewer.
3. Set env vars on this client's Vercel project: `GOOGLE_SERVICE_ACCOUNT_EMAIL`,
   `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY` (base64-encoded), `GOOGLE_SHEET_ID`.
4. Copy `config/client.config.example.ts` → `config/client.config.ts`, fill in `name`/`slug`.

## Data freshness

Reads are cached for `DEFAULT_REVALIDATE_SECONDS` (300s, set in `source.ts`) via Next's
`fetch` `revalidate` option — the dashboard does **not** see a new number the instant
the Sheet changes, it's stale for up to 5 minutes. That's intentional for a monthly-close
cadence; lower the constant if a client genuinely needs near-real-time reads, but do it in
one place (`source.ts`), not per-fetch.

## Errors

Both `getValues()` (Sheets) and any future QBO calls should throw a clear error rather
than silently returning empty/zero data — a missing tab or renamed column must surface as
a visible "data source misconfigured" state in the UI, never as quietly-wrong numbers.
